# 桌宠形象化与动画系统设计

> 窗口：ARCH-08
> 日期：2026-07-24
> 状态：用户已确认，待实现
> 本文取代 `2026-07-15-pet-customization-design.md` 中“桌宠形象化”的技术细节；共享养成进度的皮肤方向保持不变。

---

## 1. 目标与范围

当前宠物使用 Emoji 占位。本阶段将其替换为原创的高清手绘 Q 版角色，并建立可扩展的动画与皮肤协议。

首发角色为“暖色奶油星团”：

- 基础物种：原创团子兽，不采用普通猫狗；
- 视觉：圆润、低重心、短肢体，确保约 200×200 像素窗口中的轮廓识别度；
- 幻想特征：额头星纹、尾端星光等少量星灵元素；
- 首期只有一个 `base` 形态；
- 架构预留幼年、成长、进化等多形态；
- 首期不实现用户上传照片、AI 生成、外部素材包导入、Live2D 或 3D。

长期目标是让画师素材、用户手工素材和 AI 生成素材都转换为同一种标准皮肤包。核心应用不依赖具体素材来源或 AI 服务。

---

## 2. 技术决策

### 2.1 素材源：独立透明帧

首期使用独立透明 PNG/WebP 动画帧和 JSON 清单，不要求美术直接交付整张精灵图。

理由：

- 适合画师逐帧修改；
- 适合未来 AI 按动作生成和修正；
- 适合用户素材包导入；
- Electron 本地资源不存在网络请求压力；
- 后续可在构建或导入阶段自动打包图集，不改变源素材协议。

### 2.2 运行时：Canvas 帧渲染

- 启动时预加载并解码当前皮肤素材；
- 使用单个透明 Canvas 按时间绘制；
- 根据 `devicePixelRatio` 设置内部画布尺寸，CSS 尺寸保持现有响应式缩放；
- 使用 `requestAnimationFrame` 调度，但按时间差决定帧序号，不按回调次数计帧；
- 空闲且当前画面无需更新时停止高频重绘；
- 水平移动通过 Canvas 变换翻转，素材无需准备左右两套。

不使用反复修改 `<img src>` 作为正式方案，避免切帧闪烁和 DOM 状态分散。

### 2.3 渲染器边界

`pet.js` 只发出语义动作：

```js
animationController.play('walk')
animationController.play('eat')
```

动画控制器不处理养成业务；Canvas 渲染器不处理动作优先级。

```text
点击 / 喂食 / 移动 / 心情变化
              ↓
           pet.js
              ↓
       AnimationController
              ↓
        FrameRenderer
              ↓
      当前皮肤素材清单
```

未来若接入 Live2D 或 3D，只增加兼容同一动作语义的渲染器，不改 `pet.js` 的业务调用。

## 3. 组件职责

| 组件 | 职责 | 不负责 |
|------|------|--------|
| `pet.js` | 把点击、喂食、移动、心情等业务行为映射为语义动作 | 图片加载、帧序号计算 |
| `AnimationController` | 动作优先级、打断、恢复、循环/单次状态、当前朝向 | Canvas 绘制、PetState 业务 |
| `FrameRenderer` | 素材预加载、时间驱动换帧、Canvas 绘制、高 DPI、水平翻转 | 动作选择、养成状态 |
| 皮肤清单加载器 | 读取与校验 `pet.json`、解析形态和动作资源 | 播放动画 |
| 形态选择纯函数 | `skin + level → formId` | 修改 PetState |

面板角色卡只读取皮肤清单中的 `portrait`，不启动桌宠动画。

## 4. 标准动作与状态机

首期标准动作：

| 动作 | 播放方式 | 触发条件 |
|------|----------|----------|
| `idle` | 循环 | 默认 |
| `walk` | 循环 | 自动移动期间 |
| `eat` | 单次 | 喂食成功 |
| `happy` | 单次 | 高心情互动、升级或正向反馈 |
| `sad` | 单次 | 心情低于 30 且静止时，每隔 2–5 分钟随机触发一次 |
| `interact` | 单次 | 用户单击 |
| `sleep` | 循环 | 连续 10 分钟无用户互动，且当前没有移动或一次性动作 |

默认优先级：

```text
eat > happy/interact > walk > sad/sleep > idle
```

规则：

- 高优先级动作可打断低优先级动作；
- 一次性动作结束后根据实时业务状态重新选择基础动作；
- 不机械恢复被打断动作，避免喂食结束后恢复已经失效的 `walk`；
- 双击打开面板时，不等待 `interact` 播放完成；
- `sad` 只在心情低于 30 且宠物静止时调度；以普通 `idle` 为主，每隔 2–5 分钟随机播放一次；
- 移动或其他动作期间不触发 `sad`；
- 缺失动作按清单配置回退，最终回退到 `idle`；
- 具体优先级写入集中配置，不散落在业务条件中。

## 5. 皮肤包协议

### 5.1 目录

```text
cream-star/
├── pet.json
├── portrait.webp
└── forms/
    └── base/
        ├── idle/
        ├── walk/
        ├── eat/
        ├── happy/
        ├── sad/
        ├── interact/
        └── sleep/
```

### 5.2 清单字段

`pet.json` 至少描述：

- 协议版本；
- 皮肤 ID、名称、作者和皮肤版本；
- 立绘路径；
- 默认形态及形态列表；
- 形态解锁等级；
- 每个动作的帧列表、帧率、循环方式；
- 角色锚点、显示比例和安全边界；
- 是否允许水平翻转；
- 缺失动作回退目标。

示意：

```json
{
  "schemaVersion": 1,
  "id": "cream-star",
  "name": "奶油星团",
  "portrait": "portrait.webp",
  "defaultFormId": "base",
  "forms": {
    "base": {
      "unlockLevel": 1,
      "scale": 1,
      "anchor": { "x": 0.5, "y": 0.92 },
      "flipX": true,
      "animations": {
        "idle": {
          "fps": 8,
          "loop": true,
          "frames": ["forms/base/idle/001.webp"]
        }
      }
    }
  }
}
```

正式实现前需为清单定义独立校验器和测试样例；上例不作为未讨论字段的扩展授权。

### 5.3 美术规范

- 推荐源画布：512×512；
- 每帧透明背景、统一画布尺寸；
- 角色脚底锚点保持一致，禁止播放时位置抖动；
- 推荐 8–12 fps；
- `idle` 约 4–8 帧；
- `walk` 约 6–10 帧；
- 一次性动作约 6–12 帧；
- 优先无损 WebP，导入层兼容 PNG；
- 角色、尾端星光和阴影不得越过配置的安全边界。

## 6. 状态与成长

PetState 首期新增：

```js
activeSkinId: 'cream-star',
ownedSkinIds: ['cream-star']
```

动画帧、当前播放时间、动作队列和朝向属于运行时状态，不持久化。

形态由纯函数计算：

```text
activeSkinId + level → activeFormId
```

首期只有 `base`。未来可在清单中增加：

```text
Lv.1–9   → base
Lv.10–19 → grown
Lv.20+   → evolved
```

等级、经验、心情、饱腹、亲密度、金币和库存始终属于同一套养成状态；换皮肤或进化不复制这些数据。

新增持久化字段时，必须同步主进程默认状态和完整快照保护逻辑，避免 PetState 整体覆盖写盘丢字段。

## 7. 与现有功能的集成

- 保留现有 `window:move` IPC 和 CSS 原生拖拽；
- Canvas 宠物本体为 `-webkit-app-region: no-drag`，外围仍提供拖拽区域；
- 单击继续触发气泡、经验和心情，并播放 `interact`；
- 双击继续切换面板；
- 喂食成功播放 `eat`；
- 自动走动期间播放 `walk`；
- 低心情可进入 `sad`；
- 连续 10 分钟没有点击、喂食或拖拽，且当前没有自动走动或一次性动作时进入 `sleep`；
- 点击、喂食或拖拽会立即退出 `sleep`；
- 面板角色卡使用当前皮肤 `portrait`；
- 气泡和番茄钟浮标继续作为 Canvas 上方的独立 DOM 层；
- 四档窗口缩放和高 DPI `setBounds` 机制保持不变。

本阶段不新增跨模块 EventBus 事件。若实现时发现必须新增事件，先由架构窗口确认并同步 `events.js` 与 `docs/events.md`。

## 8. 容错与安全

- 默认皮肤加载失败：使用内置静态兜底图；
- 静态兜底图失败：回退 Emoji；
- 非关键动作缺失：按清单回退至 `idle`；
- 单帧损坏：跳过该帧；动作无可用帧时走动作回退；
- 动画播放异常：恢复 `idle`，不得影响移动、喂食和窗口切换；
- 切换皮肤时先校验、预加载，再原子替换；
- 未知清单字段忽略，使用 `schemaVersion` 管理兼容性；
- 未来外部皮肤路径必须限制在皮肤包内部；
- 外部包限制文件类型、尺寸和总大小；
- 用户素材只作为数据读取，禁止执行其中的脚本。

## 9. 测试与验收

### 9.1 自动测试

- 动作优先级、打断和结束恢复；
- `skin + level → formId`；
- 清单字段校验、路径安全和版本兼容；
- 动作缺失与损坏帧回退；
- 时间驱动帧计算、循环与单次结束；
- 水平翻转和高 DPI 画布尺寸；
- 资源预加载成功/失败及原子切换。

### 9.2 Electron 手工验收

- 启动显示奶油星团，不再显示 Emoji；
- 待机、走路、进食、开心、低落、互动、睡觉动作符合触发条件；
- 单击、双击、拖拽、自动走动、喂食、缩放、面板切换不退化；
- 气泡和番茄钟浮标层级正确；
- 面板立绘与桌宠皮肤一致；
- 高 DPI 下清晰且窗口尺寸不漂移；
- 素材失败时能够逐级降级，不导致白屏或宠物消失。

### 9.3 性能验收

- 空闲且画面无需更新时不持续高频重绘；
- 只预加载当前皮肤和当前形态；
- 切换页面后无遗留动画循环；
- 连续运行时无明显内存增长。

## 11. 参考调研

- Shimeji 类桌宠仍以动作图片集为核心，适合轻量和用户自定义；
- Live2D Cubism Web SDK 使用 WebGL，适合高表现力 2D，但模型制作和授权复杂度更高；
- Spine Web 运行时适合骨骼动画，但当前团子兽和 200×200 常驻窗口不需要首期引入；
- Desktop Mate 代表高品质 3D 路线，表现力强，但资源、引擎和性能成本不符合当前阶段。

结论：首期采用独立帧协议和 Canvas 渲染；未来通过渲染器边界升级，而不是提前引入重型运行时。
